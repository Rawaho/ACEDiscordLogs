Mag-nus 16-Feb-18 08:54 AM
So, I'd like to get some feedback regarding a database design question.

The question is: When a new object is created, should all of the weenie properties be copied (duplicated) into that object? Or, should only values that differ from the weenie be stored in the objects database record?

In retail, all of the values were copied over. The example is the hoary robe. When they changed the weenie, all the existing instances of the robe remained in the previous state.

Alternatively, if we only store the values in the objects db record that differ from the weenie, changing a weenie would end up changing every instance in the world. This might be nice for special events, or custom mods. It would also greatly reduce DB usage.

This would prevent us from changing an existing weenie for newly created objects while keeping existing objects as they were. If a server operator wanted to change a weenie only moving forward, they'd really need to retire that weenie from the loot table and create a new weenie to replace it with the desired values.
I'd like to add that the WorldObject has a reference to the weenie, so the weenies base properties are always available to the world object regardless of which design we go with.
