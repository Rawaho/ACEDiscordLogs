Amauxi 01-Dec-17 01:19 PM
Stairs, sliding, friction, things like that would be of concern if you were trying to use something else
Much easier to use the real thing
Animations are a bit complicated but there's 3 states really
forward state, sidestep state, turning state, then of course theres the animation sequences themselves
Sometimes you can have a weird blend happening like doing a lifestone animation and side stepping at the same time
Spellcasting and moving at the same time
The high bits on the animation "command" are the type of animation or purpose
the low bits are the index of the animation
For example, the 0x1000000 flag (CM_Action) is something you'll only see in the queued list of sequences
shouldn't see the flag in the forward/sidestep/turn commands
but it's the low bits that are the actual index itself
MotionWave = 0x13000087, is actually CMAction|CMChatEmote|CMMappable|0x87

Amauxi 01-Dec-17 01:31 PM
The server in turn digests, processes it, spits out an"interpolated state"
The client receives the interpolated state for all objects in the game that are moving
That interpolated state is the same one you see in 0xF745 (CreateObject)
That's just telling the client what the current animation state is when it first receives the object
Anytime it changes, the server tells the client
If the position changes too much, or enough time passes and the position changed, the server also spits out exact positional data as well
So there's the "raw motion state" created by the client that is just a reflection of the keyboard/emote input, that is sent to the server, but the only thing clients ever receive are "interpolated motion states"(edited)

Amauxi 02-Dec-17 05:27 AM
The MovementManager, MotionTable, CMotionInterp, that whole set of things is what you'll need for proper movement/animation blending states
MovementManager would be very useful to reverse engineer
You already know by the network protocol there are commands like 'turn to this object' 'move to this object' etc. except what you may not know, although it is intuitive once you do, is that the server is executing the same movement code the client is, so when the server says 'run towards this object' to the client, the server also begins executing the same logic the client does in order to animate it in that direction, the server will then only update the client periodically on the position of the object, but the server knows the client is already animating it to move in the right manner
So for monsters to work proper, you not only need physics, you need this movement code
